Build & Deployment Pipeline

1
Build React frontend
npm run build inside frontend/. Vite bundles the React app and outputs it directly to backend/src/main/resources/static/ — inside the Spring Boot source tree, ready to be packaged into the jar.
Vite 5React 18Tailwind CSS
2
Package Spring Boot jar
mvn package inside backend/. Maven packages the Spring Boot app plus the React files from src/main/resources/static/ into a single fat jar: backend/target/payflow.jar. Spring Boot's ResourceHttpRequestHandler serves the frontend at /.
Maven 3Spring Boot 3.5JDK 21
3
cf push
CF CLI reads manifest.yml, uploads the jar, stages it with java_buildpack_offline (installs JDK 21), binds the three service instances, and starts the app. CF assigns a dynamic PORT and registers the route with Gorouter.
java_buildpackDiegoGorouter
4
java-cfenv-boot wires the services
On startup, java-cfenv-boot reads VCAP_SERVICES (injected by CF) and overrides Spring's datasource, RabbitMQ, and Redis connection properties. The app connects to the managed services with zero config changes.
java-cfenv-bootVCAP_SERVICESAuto-configuration
✓
App is live
Gorouter registers the route. The React SPA is served at /. REST API is at /api/*. WebSocket at /ws. Health check at /actuator/health. All traffic flows through a single CF route — no CORS, no separate frontend app.
Single routeNo CORSHealth check

Payment Lifecycle on CF

1
User submits payment
Browser calls POST /api/payments. Gorouter forwards to the CF app container. On CF, relative URLs work because the frontend is served from the same app — no CORS, no proxy needed.
GorouterRESTSame-origin
2
Saved PENDING → broadcast via WebSocket
Transaction persisted to CF-managed PostgreSQL. WebSocket event sent to /topic/transactions. SockJS connects to window.location.origin + /ws — automatically correct for any CF route.
CF PostgreSQLWebSocket
3
Published to CF-managed RabbitMQ
PaymentProducer publishes to payflow.exchange. The exchange, queue, and DLQ are declared by Spring's AMQP auto-configuration using the CF binding credentials. All durable — survives app restarts.
CF RabbitMQpayment.queueDLQ
4
Consumer → Fraud detection (CF Redis)
PaymentConsumer picks up the message, runs the fraud rules. Rapid succession rule uses INCR rapid:<accountId> on the CF-managed Redis instance. TTL 60s — no cleanup job, no stale state.
CF RedisINCR + EXPIRE
✗
FLAGGED
FraudAlert saved to CF PostgreSQL. Broadcast on /topic/fraud-alerts. Badge counter increments on all browsers.
✓
COMPLETED
Balances updated in CF PostgreSQL. Broadcast on /topic/transactions. Chart and KPI cards refresh everywhere.
